# 실제 배포 진행 계획

## 1. 목표

이 문서는 `나중에 실제 운영 서버에 올릴 때` 무엇을 어떤 순서로 진행할지
미리 잠가두는 배포 런북이다.

핵심 목적은 세 가지다.

- 배포 당일에 즉흥적으로 판단하지 않기
- 검증 포인트와 중단 기준을 미리 정하기
- 실패 시 빠르게 롤백할 수 있게 하기

---

## 2. 배포 전략

### 2.1 기본 전략

- v1은 `단일 Linux VM + systemd timer + nginx`로 배포
- 새 서버에서 충분히 검증한 뒤 공개 전환
- 공개 전환은 `도메인 연결` 또는 reverse proxy 전환으로 수행
- 첫 공개 시에는 `조용한 소프트 오픈`을 기본값으로 함

### 2.2 배포 원칙

- 한 번에 모든 기능을 열지 않는다.
- `크롤링 성공 -> 정적 웹 생성 -> 외부 노출` 순으로 단계 분리
- 공개 전에 최소 1일 이상 스케줄 실행 검증
- 롤백은 `정적 웹 이전 버전 + SQLite 백업` 기준으로 단순화

---

## 3. 사전 결정 사항

배포 전에 아래 항목이 확정되어 있어야 한다.

### 3.1 인프라

- VM 제공자 확정
- 서버 리전 확정
- 서버 OS 확정
- 도메인 또는 서브도메인 확정

### 3.2 운영 기준

- 실제 운영 URL
- 운영자 연락 이메일 또는 제보 채널
- 텔레그램/슬랙 발송 여부
- 공개 범위
  - 내부 테스트만
  - 제한 공개
  - 대외 공개

### 3.3 배포 승인 기준

- 최근 연속 3회 배치 성공
- `pipeline_status.json` 기준 오류 없음
- 홈/뉴스/정책/허브 페이지 정상 렌더링
- 허브/정책 메뉴에 최신 데이터 반영 확인

---

## 4. 단계별 진행표

## 4.1 D-7 ~ D-3: 사전 준비

### 해야 할 일

- `python scripts/deployment_readiness.py`로 입력값 누락 확인
- 서버 생성
- Linux 계정/권한 구조 준비
- 방화벽 열기 (80/443)
- 프로젝트 코드 배치
- Python 가상환경 설치
- nginx 설치
- 환경변수 파일 생성
- systemd service/timer 등록
- 자동 백업 timer 등록

### 완료 기준

- 서버에서 수동 배치 1회 성공
- `web/index.html`이 nginx로 노출됨
- `pipeline_status.json` 접근 가능

### 중단 조건

- 크롤링이 서버 환경에서 전혀 동작하지 않음
- 파일 권한 문제로 배치 출력이 생성되지 않음
- nginx 기본 정적 서빙이 안 됨

---

## 4.2 D-2 ~ D-1: 스케줄 검증

### 해야 할 일

- 실제 스케줄(09:00 / 15:00 / 21:00) 또는 임시 단축 스케줄로 자동 실행 검증
- 최소 3회 연속 성공 확인
- 로그 파일 회전 확인
- lock 파일이 정상 제거되는지 확인
- 정적 HTML이 매 실행마다 갱신되는지 확인
- 자동 백업이 1회 이상 성공하는지 확인

### 점검 항목

- `systemctl status youth-together-pipeline.timer`
- `systemctl status youth-together-pipeline.service`
- `systemctl status youth-together-backup.timer`
- `output/pipeline_status.json`
- `output/scheduler_logs/`
- 최신 기사 날짜 기준

### 완료 기준

- 배치 연속 성공 3회 이상
- 백업 1회 이상 성공
- `last_task_result = 0`
- 다음 실행 시각이 정상 표시
- 웹 페이지 최신 반영 확인

---

## 4.3 D-1: 콘텐츠 품질 점검

### 해야 할 일

- 뉴스 페이지 최신 기사 검토
- 정책 페이지 공식 발표 검토
- 허브 페이지 정부 거버넌스 분류 확인
- 중복 기사 과다 노출 여부 확인
- 날짜 표기 / 업데이트 시각 표기 확인

### 운영 체크

- 너무 뻔한 반복 기사 노출 여부
- 허브 오분류 여부
- 공백 섹션(특히 지역 거버넌스) 안내 문구 적절성
- 사이트 전반적인 첫인상 점검

### 완료 기준

- 공개 직후 신뢰를 해칠 정도의 오분류 없음
- 메인 홈에서 핵심 정보가 3초 안에 읽힘

---

## 4.4 D-Day: 공개 전환

### 순서

1. 마지막 수동 배치 1회 실행
2. `pipeline_status.json` 성공 확인
3. 메인 페이지/뉴스/정책/허브 수동 확인
4. 도메인 또는 reverse proxy 연결
5. 외부 접속 확인
6. HTTPS 확인
7. robots / 기본 메타 / 문의 경로 확인

### 공개 직후 30분 체크

- 홈 접속 속도
- 404 여부
- CSS/폰트 깨짐 여부
- 외부 링크 정상 이동 여부
- 모바일 렌더링 기본 확인

### 공개 직후 2시간 체크

- 로그 폭주 여부
- CPU/메모리 과점유 여부
- 크롤링 타임아웃 여부
- Telegram/Slack 발송이 켜져 있다면 발송 정상 여부

## 4.5 D+1 ~ D+7: 초기 운영 안정화

### 매일 확인할 것

- 마지막 성공 시각
- 다음 배치 예약 시각
- 최종 기사 수
- 허브 정부/지역 분류 상황
- 실패 로그 여부

### 첫 주에 특히 볼 것

- 특정 소스가 자주 비는지
- 지역 피드가 계속 비어 있는지
- 정책 페이지가 실제 최신 발표를 놓치지 않는지
- 홈 카드 업데이트 문구가 현실과 맞는지

## 5. 롤백 계획

## 5.1 롤백이 필요한 상황

- 메인 페이지가 깨져 노출 불가
- 배치가 연속 실패해 최신 데이터 반영 불가
- 잘못된 크롤링/분류로 신뢰 문제가 생김
- 서버 성능 문제로 서비스가 불안정함

## 5.2 롤백 방식

### 1차 롤백

- nginx가 바라보는 `web/`를 이전 정상 버전으로 교체
- 최신 배치 실행 중지
- 홈 공지나 운영 채널로 점검 안내

### 2차 롤백

- SQLite 백업 복원
- 최근 코드 버전 이전 커밋으로 되돌려 재배포
- systemd timer 일시 중지

### 롤백 준비물

- 배포 전 `db/articles.db` 백업
- 배포 전 `web/` 스냅샷
- 최근 정상 실행 시점의 `pipeline_status.json`

## 8. 다음 실제 작업 후보

실제 배포 직전에는 이 순서가 가장 자연스럽다.

1. VM 공급자 1개 확정
2. 도메인/서브도메인 확정
3. 서버 계정/경로 기준으로 `deploy/` 템플릿 실제 값 치환
4. 배포 리허설 1회
5. 하루 동안 자동 배치 관찰
6. 공개 전환

이 문서는 `실제 배포가 가까워졌을 때` 바로 체크리스트로 사용한다.
