> **문서 목적**: Claude Code에서 구현 시 참조할 계획서  
> **작성일**: 2026-03-28  
> **상태**: 초안 (리뷰 대기)

## 1. 작업 컨텍스트

### 1.1 배경 및 목적

청년 이슈에 관한 정보가 여러 매체에 파편화되어 있어, 청년정책 활동가·연구자·관심 시민이 현황을 파악하고 정책을 제안하는 데 비효율이 발생한다. 이 플랫폼은 **"청년 이슈의 수집 → 정제 → 전달 → 활용"** 전 과정을 자동화하여, 누구나 무료로 접근할 수 있는 공공 도구로 운영한다.

**핵심 가치**: 수익화가 아닌 공공 기여. 트래픽 확보 이후 광고 등 최소한의 지속 가능성 모델 검토.

**제품 운영 원칙**:
- 빠른 포괄성을 우선하되, 동일 주제 반복 노출과 편향을 구조적으로 줄인다.
- 청년정책 활동가뿐 아니라 라이트 이용자와 연구자도 함께 쓰는 공공 정보 인프라를 지향한다.
- 정책 제안서 도구는 활동가의 사고를 돕는 보조도구여야 하며, 사용자의 판단을 대체하는 자동 대필 도구가 되어서는 안 된다.
- 로컬 이슈는 전국 이슈의 하위 범주가 아니라 독립적으로 존중받아야 할 정보 축으로 다룬다.

### 1.1.1 핵심 사용자상

| 사용자군 | 주요 행동 | 제품에서 얻어야 하는 가치 |
|----------|----------|--------------------------|
| 라이트 이용자 | 메인 페이지에서 빠르게 기사와 브리핑 확인 | 청년정책·청년현황 정보를 짧은 시간 안에 직관적으로 파악 |
| 헤비 사용자(활동가) | 기사 확인 + 정책 제안서 도구 활용 | 정책 제안 활동의 근거 수집, 논점 정리, 작성 품질 향상 |
| 연구자 | 기사 흐름과 향후 커뮤니티 반응 참고 | 청년 현장의 반응과 담론 흐름을 추적 |

### 1.1.2 실패 조건과 성공 기준

**가장 치명적인 실패**:
- 정보가 특정 논조나 특정 매체군에 과도하게 편향되는 것
- 같은 이슈의 유사 기사만 반복 노출되어 정보 효율이 떨어지는 것
- 새로운 정책 발표나 신규 이슈를 하루 안에 포착하지 못하는 것

**성공한 상태의 기준**:
- 메인 페이지 첫 화면만으로도 청년정책 브리핑과 핵심 현황을 빠르게 파악할 수 있다.
- 공식 청년정책 발표는 빠짐없이 포함하되, 유사 기사 묶음을 정리해 반복 피로를 줄인다.
- 로컬 이슈와 전국 이슈가 한 화면 안에서 균형 있게 공존한다.

### 1.2 범위

| 포함 (v1) | 미포함 (향후 확장) |
|-----------|-------------------|
| 뉴스 기사 수집·요약·분류 | 커뮤니티/포럼 기능 |
| 유튜브 콘텐츠 수집·정리 | 사용자 맞춤 알림 설정 |
| 지역별 이슈 분류 탭 | 정책 효과 분석 대시보드 |
| 텔레그램/슬랙 봇 다이제스트 | 다국어 지원 |
| 웹 프론트엔드 (열람·검색) | 정책 제안서 협업 편집 |
| 정책 제안서 템플릿·가이드라인 | LLM 대화형 제안서 작성 |
| 공식 정책 발표 우선 브리핑 | 연구자용 커뮤니티 반응 탐색 기능 (v2) |

### 1.2.1 사이트 기능 구조 (개정)

사이트의 전반적 기능 구조는 다음 5개 축으로 재편한다.

1. `청년 관련 뉴스 모아보기 + 데일리 하이라이트`
   - 청년 관련 기사 수집
   - 일일 핵심 기사 하이라이트 요약
   - 카테고리 기반 탐색
2. `최근 발표된 정책 모아보기`
   - 정부발 공식 발표 중심
   - 부처/기관 단위 탐색
   - 최근 발표 순 정렬
3. `청년활동가 허브`
   - 정부 거버넌스 활동 소식
   - 지역 거버넌스 활동 소식
   - 커뮤니티(정책 제안, 협업 제안)
4. `정책제안서 작성 도구`
   - 정책 서칭 주요 사이트 모음
   - AI 도구 활용법
   - 정책제안서 검토 요청
5. `기타 운영자 연락하기`
   - 제보, 협업, 문의 채널
   - 운영 안내 및 피드백 접수

### 1.3 입출력 정의

**입력**:
- 뉴스 소스: 주요 언론사 RSS 피드 + 네이버 뉴스 검색 크롤링 (보완용)
- 유튜브: YouTube Data API v3 키워드 검색 결과
- 수집 키워드: "청년" + 카테고리별 세부 키워드 (아래 키워드 체계 참조)
- 정책 제안서 템플릿: 사전 제작된 문서 파일

**출력**:
- 일일 다이제스트: 카테고리별 핵심 기사 10건 이내 (제목, 3줄 요약, 원문 링크, 카테고리, 지역 태그, 공식발표 여부)
- 유튜브 큐레이션: 주요 영상 목록 (제목, 채널명, 요약, 링크)
- 웹 페이지: 전체 아카이브 + 카테고리/지역 필터 검색
- 봇 메시지: 텔레그램/슬랙 일일 발송
- 도구 페이지: 정책 제안서 템플릿, 가이드라인, 참고자료 링크

### 1.4 키워드 체계

| 카테고리 | 수집 키워드 (예시) | 비고 |
|----------|-------------------|------|
| 정책 오피셜 | 청년정책, 청년 일자리, 청년 주거, 청년 복지, 청년수당, 정부 발표, 부처 브리핑 | 정부 원문 우선 |
| 청년은 지금 | 청년 실태, 청년 통계, 청년 고용, 청년 부채, 청년 주거난, 청년 삶 | 청년 현실·근황·통계 우선 |
| 논평·기고 | 청년 칼럼, 청년 담론, 청년 세대 분석, 청년 논평 | 과잉 비중 방지 필요 |
| 지역 이슈 | 지방소멸, 지역 청년, 청년 유출, 로컬크리에이터, 지역 청년정책 | 광역시도 태그 자동 추출 |

### 1.5 제약조건

- **비용**: LLM API 호출 비용 최소화 (배치 처리, 요약 대상 사전 필터링)
- **법적**: 기사 본문 전문 저장 금지 → 제목 + 요약 + 원문 링크만 보관
- **기술**: 네이버/다음 크롤링 시 robots.txt 준수, rate limiting 적용
- **품질**: 노이즈 기사 필터링 (광고성 기사, 단순 홍보 등 제거)
- **신속성**: 공식 정책 발표와 주요 신규 이슈는 발표 후 24시간을 넘기지 않도록 정기 업데이트
- **편집 금기**: 과도한 개인화, 감정적 푸시, 조회수 중심 편집, 정치 성향 오해를 부르는 랭킹, 과도한 자동 대필 금지

### 1.6 용어 정의

| 용어 | 정의 |
|------|------|
| **다이제스트** | 하루치 수집 결과를 LLM이 선별·요약한 최종 결과물 (10건 이내) |
| **지역 태그** | 기사 본문에서 추출된 17개 광역시도 단위 지역 정보 |
| **노이즈 기사** | 청년 키워드를 포함하지만 실질적으로 청년 이슈와 무관한 기사 |
| **큐레이션** | 수집된 콘텐츠 중 품질·관련성 기준으로 선별하는 과정 |
| **대표 기사** | 동일 이슈로 묶인 기사군에서 메인 카드로 노출되는 기준 기사 |
| **공식 발표** | 국무조정실 및 주요 정부부처, 광역지자체 등 공공기관이 직접 발표한 원문 자료 |

## 2. 워크플로우 정의

### 2.1 전체 흐름 개요

```
[스케줄러: 1일 3회 트리거 (09:00 / 15:00 / 21:00, Asia/Seoul)]
        │
        ▼
┌─────────────────┐
│  Step 1: 수집    │  ← 스크립트 처리
│  RSS + 크롤링    │
│  + YouTube API   │
└────────┬────────┘
         │ 수집된 원본 목록 (JSON)
         ▼
┌─────────────────┐
│  Step 2: 중복제거 │  ← 스크립트 처리
│  + 기본 필터링    │
└────────┬────────┘
         │ 필터링된 목록
         ▼
┌─────────────────┐
│  Step 3: 분류    │  ← LLM 판단
│  카테고리 + 지역  │
│  + 노이즈 판별   │
└────────┬────────┘
         │ 분류 완료 목록
         ▼
┌─────────────────┐
│  Step 4: 선별    │  ← LLM 판단
│  중요도 스코어링  │
│  상위 10건 선정   │
└────────┬────────┘
         │ 최종 선별 목록
         ▼
┌─────────────────┐
│  Step 5: 요약    │  ← LLM 생성
│  3줄 요약 생성    │
└────────┬────────┘
         │ 요약 완료 데이터
         ▼
┌──────────────────────────────┐
│  Step 6: 발행                 │  ← 스크립트 처리
│  DB 저장 + 웹 갱신 + 봇 발송  │
└──────────────────────────────┘
```

### 2.2 단계별 상세

#### Step 1: 콘텐츠 수집

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 (`collect-news`, `collect-youtube`) |
| **입력** | RSS 피드 URL 목록, 네이버 뉴스 검색 API, YouTube Data API v3 |
| **출력** | `/output/step1_raw_articles.json`, `/output/step1_raw_youtube.json` |
| **처리 방식** | RSS 파싱 → 제목, URL, 발행일, 본문 일부(리드 200자) 추출. 네이버 뉴스 검색으로 RSS 누락 기사 보완. YouTube는 키워드별 최신 영상 검색. |
| **성공 기준** | 최소 1개 소스에서 데이터 수집 성공. 각 기사에 title, url, published_date 필드 존재 |
| **검증 방법** | 스키마 검증 (필수 필드 존재 + 타입 체크) |
| **실패 시** | 개별 소스 실패 → 스킵 + 로그 (다른 소스로 계속). 전체 실패 → 에스컬레이션 (Slack 알림) |

**RSS 소스 전략**:
- 1차 소스 (필수): 연합뉴스, YTN, 한겨레, 경향신문, 한국일보 등 중앙 매체 RSS
- 1차 공식 소스 (필수): 국무조정실 및 주요 정부부처의 정책 브리핑/보도자료 페이지 또는 RSS
- 2차 소스 (보완): 네이버 뉴스 검색 API로 지역 매체, 전문 매체 커버리지 보완
- 소스 우선순위: 공식 발표 원문 > 중앙 대표 기사(연합뉴스/YTN 중심) > 지역 대표 기사(지역명 포함 매체 중심)
- 업데이트 원칙: 일반 뉴스와 공식 정책 발표 소스는 1일 3회 정기 수집을 기본으로 하며, 공식 정책 발표는 같은 주기 안에서 우선 모니터링하여 발표 후 24시간 내 반영을 목표로 한다.

#### Step 2: 중복 제거 + 기본 필터링

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 (`dedup-filter`) |
| **입력** | Step 1 산출물 |
| **출력** | `/output/step2_filtered.json` |
| **처리 방식** | URL 정규화 후 중복 제거 → 제목 유사도(편집 거리) 기반 유사 기사 묶음 → 블랙리스트 도메인 제거 → 키워드 기본 필터(제목+리드에 관련 키워드 1개 이상 포함 확인) → 유사 기사군별 대표 기사 후보 계산 |
| **성공 기준** | 중복률 0%, 필터 후 최소 5건 이상 잔존 |
| **검증 방법** | 규칙 기반 (중복 URL 0건 확인, 최소 건수 확인) |
| **실패 시** | 5건 미만 잔존 → 키워드 필터 완화 후 자동 재시도 (1회) |

**중복 기사 대표 선정 규칙**:
- 정부 보도자료를 기반으로 다수 언론이 유사 기사를 작성한 경우, 정부 원문을 최우선 대표 기사로 둔다.
- 언론 기사 보존이 필요할 경우 중앙 대표 1건(연합뉴스 또는 YTN 우선) + 지역 대표 1건(지역명 포함 매체 우선)만 남긴다.
- 뻔한 후속 반복 기사, 단순 재가공 기사, 제목만 바꾼 동일 논지 기사는 유사 기사 수 집계에만 반영하고 메인 목록에서는 제외한다.

#### Step 3: 카테고리 분류 + 지역 태그 + 노이즈 판별

| 항목 | 내용 |
|------|------|
| **처리 주체** | **LLM 판단** (에이전트) |
| **입력** | Step 2 산출물 (제목 + 리드 200자) |
| **출력** | `/output/step3_classified.json` |
| **LLM이 판단하는 것** | ① 4개 카테고리(`정책 오피셜`, `청년은 지금`, `논평·기고`, `지역 이슈`) 중 해당 카테고리 분류, ② 지역 정보가 있으면 17개 광역시도 중 태그 부여 (없으면 `전국`), ③ 노이즈 여부 판별 (광고성, 청년과 무관한 기사 제거), ④ 공식발표 여부와 대표 기사 후보 적합성 판단 |
| **성공 기준** | 모든 기사에 카테고리 1개 이상 부여. 노이즈 판별 완료. JSON 스키마 준수 |
| **검증 방법** | 스키마 검증 (필수 필드) + LLM 자기 검증 (분류 근거 1줄 생성하여 명백한 오분류 자체 교정) |
| **실패 시** | 스키마 미준수 → 자동 재시도 (최대 2회). 분류 불확실 → "미분류" 태그 부여 후 계속 |

**분류 프롬프트 설계 핵심**:
- 제목+리드만으로 판단 (본문 전체 불필요 → 비용 절감)
- 배치 처리: 한 번의 LLM 호출에 최대 20건 묶어서 처리
- 출력 포맷: 구조화된 JSON으로 강제
- 청년층이 주요 대상이 되는 기사라면 폭넓게 포함하되, 카테고리 오분류와 노이즈 판별은 보수적으로 수행
- 공식 청년정책 발표, 국무조정실·주요 부처 발표, 광역지자체의 청년정책 브리핑은 노이즈 판별에서 예외 처리

#### Step 4: 중요도 선별

| 항목 | 내용 |
|------|------|
| **처리 주체** | **LLM 판단** (에이전트) |
| **입력** | Step 3 산출물 (노이즈 제거 후) |
| **출력** | `/output/step4_selected.json` |
| **LLM이 판단하는 것** | 중요도 스코어링 기준: ① 청년 현실·근황·통계에 대한 설명력, ② 정책 영향력 (새 법안, 예산, 제도 변경), ③ 정책 안내·홍보로서의 실용성, ④ 지역 고유성 (전국 뉴스에서 잘 다루지 않는 지역 이슈), ⑤ 시의성 (신규 이슈 > 후속 보도). 상위 10건 선정 + 카테고리 균형 고려 |
| **성공 기준** | 최종 선정 10건 이내. 최소 2개 카테고리 포함. 각 기사에 선정 사유 1줄 포함 |
| **검증 방법** | 규칙 기반 (건수, 카테고리 분포, 공식 정책 포함 여부) + LLM 자기 검증 (선정 결과 전체를 보고 "편향되거나 누락된 카테고리 없는지" 자체 점검) |
| **실패 시** | 카테고리 편중 → 자동 재시도 (1회, "카테고리 균형 조정" 지시 추가). 공식 정책 누락 → 경고 후 재선정 강제. 그래도 실패 → 그대로 진행 + 로그 |

**선별 우선순위 원칙**:
- `청년은 지금` 카테고리의 현실·통계·실태 기사를 가장 우선한다.
- 공식 정책 발표는 단순 홍보성 기사보다 우선하되, 정책 원문과 동일한 내용의 반복 보도는 대표 기사 체계로 축약한다.
- `논평·기고`는 보조적 축으로 유지하며, 메인 다이제스트를 과도하게 점유하지 않도록 비중 상한을 둔다.
- 로컬 이슈는 전국적 규모가 작아도 청년에게 실질적 의미가 크다면 선별 우선권을 부여할 수 있다.

#### Step 5: 요약 생성

| 항목 | 내용 |
|------|------|
| **처리 주체** | **LLM 생성** (에이전트) |
| **입력** | Step 4 선정 기사 + 원문 리드/본문 일부 |
| **출력** | `/output/step5_summarized.json` |
| **LLM이 수행하는 것** | 기사당 3줄 요약 생성. 요약 원칙: ① 핵심 사실 중심 (의견 배제), ② 숫자·날짜 등 구체적 정보 포함, ③ 원문을 읽어야 하는 이유(후킹) 포함 |
| **성공 기준** | 모든 선정 기사에 요약 존재. 요약 길이 50~150자. 원문 표현 그대로 복사하지 않음 |
| **검증 방법** | 규칙 기반 (글자 수) + LLM 자기 검증 (요약이 원문 복사가 아닌지, 사실 왜곡 없는지 확인) |
| **실패 시** | 글자 수 초과/미달 → 자동 재시도 (최대 2회). 품질 미달 → 해당 기사 요약 없이 제목+링크만 발행 |

#### Step 6: 발행

| 항목 | 내용 |
|------|------|
| **처리 주체** | 스크립트 (`publish`) |
| **입력** | Step 5 산출물 |
| **출력** | DB 저장, 웹 페이지 갱신, 텔레그램/슬랙 메시지 발송 |
| **처리 방식** | ① JSON → DB(SQLite/PostgreSQL) 저장, ② 웹 프론트엔드 정적 생성 또는 API 갱신, ③ 메인 페이지 상단 브리핑 블록 갱신, ④ 봇 API로 다이제스트 메시지 포맷팅 + 발송 |
| **성공 기준** | DB 저장 성공 + 봇 발송 성공 (HTTP 200) |
| **검증 방법** | 규칙 기반 (DB write 확인, API 응답 코드 확인) |
| **실패 시** | DB 실패 → 에스컬레이션. 봇 발송 실패 → 자동 재시도 (최대 3회, 지수 백오프) |

### 2.3 유튜브 수집 흐름 (뉴스와 병렬)

유튜브는 뉴스와 동일한 파이프라인 구조를 따르되, 수집 주기와 선별 기준이 다르다.

```
YouTube API 검색 → 중복 제거 → LLM 관련성 판별 → 요약 생성 → 발행
```

- 수집 주기: 1일 1회 (뉴스보다 빈도 낮음)
- 선별 기준: 조회수, 채널 구독자 수, 관련성을 종합 판단
- 출력: 일일 3~5건 영상 큐레이션

### 2.4 상태 전이 다이어그램

```
IDLE → COLLECTING → FILTERING → CLASSIFYING → SELECTING → SUMMARIZING → PUBLISHING → IDLE
  │                                                                           │
  │          (어느 단계든 치명적 실패 시)                                        │
  └──────────────── ERROR ◄────────────────────────────────────────────────────┘
                     │
                     └→ 에스컬레이션 (Slack 알림) → 수동 재실행 대기
```

### 3.3 에이전트 구조

**서브에이전트 1개 분리 구조**를 채택한다.

**이유**: Step 3~5(분류→선별→요약)는 모두 LLM 판단이 필요하며, 도메인 지식(카테고리 체계, 중요도 기준, 요약 원칙)이 공통된다. 이 지침을 메인 CLAUDE.md에 모두 넣으면 수집·발행 단계에서도 불필요하게 로드된다. 따라서 LLM 판단 영역을 `content-curator` 서브에이전트로 분리한다.

```
CLAUDE.md (오케스트레이터)
  │
  ├── [스크립트] collect-news    ← Step 1
  ├── [스크립트] collect-youtube ← Step 1
  ├── [스크립트] dedup-filter    ← Step 2
  │
  ├── [서브에이전트] content-curator  ← Step 3, 4, 5
  │     └── 분류 → 선별 → 요약을 순차 수행
  │
  └── [스크립트] publish          ← Step 6
```

### 3.4 서브에이전트: content-curator

| 항목 | 내용 |
|------|------|
| **이름** | `content-curator` |
| **역할** | 수집된 기사의 분류, 중요도 판단, 요약 생성을 일괄 수행 |
| **트리거** | 메인 에이전트가 Step 2 완료 후 호출 |
| **입력** | `/output/step2_filtered.json` (필터링 완료 기사 목록) |
| **출력** | `/output/step3_classified.json`, `/output/step4_selected.json`, `/output/step5_summarized.json` |
| **참조 지식** | 카테고리 체계, 17개 광역시도 목록, 중요도 스코어링 기준, 요약 작성 원칙 |
| **데이터 전달** | 파일 기반 (중간 산출물을 `/output/`에 저장) |

### 3.6 작업 단계별 처리 방식 요약

| 단계 | 에이전트 판단 | 스크립트 처리 |
|------|-------------|-------------|
| Step 1: 수집 | — | RSS 파싱, API 호출, HTML 파싱 |
| Step 2: 필터링 | — | URL 정규화, 유사도 계산, 블랙리스트 매칭 |

### 3.7 데이터 전달 패턴

- 포맷: JSON (UTF-8)
- 메인 에이전트는 파일 경로만 서브에이전트/스크립트에 전달
- 일자별 아카이브: 실행 완료 후 `/output/archive/{YYYY-MM-DD}/`로 복사

### 3.10.1 홈 화면 정보 우선순위

메인 페이지는 소개 페이지가 아니라, 각 메뉴의 최신 반영 상태를 먼저 보여주는 **업데이트 대시보드형 홈**을 기본으로 한다.

기본 순서는 다음과 같다.

1. 파이프라인 상태 + 날짜 기준 + 업데이트 주기
2. 메뉴별 최신 업데이트 카드 (`뉴스`, `정책`, `허브`, `도구`, `연락`)
3. 오늘의 하이라이트
4. 서비스 전체 구조 설명

각 메뉴 업데이트 카드에는 최소한 다음 정보가 포함되어야 한다.

- 메뉴명
- 최신 업데이트 요약 1~2개
- `최신 기사 기준` 또는 `최신 반영 기준` 시각
- `페이지 반영` 시각
- 해당 메뉴로 바로 이동하는 CTA 버튼

**설계 이유**:
- 홈은 구조 소개보다 먼저 `지금 무엇이 갱신되었는지`를 보여줘야 재방문 동기가 생긴다.
- 뉴스와 정책은 데이터성 메뉴이므로 최신 기사/발표 기준 시각을 직접 보여주는 편이 신뢰를 준다.
- 허브, 도구, 연락은 정적 메뉴처럼 보여도 실제로는 운영 상태가 반영되는 영역이므로 업데이트 기준을 명시해야 한다.

### 3.10.2 기사 카드 메타데이터

각 기사 카드는 최소한 다음 정보를 포함한다.

- 제목
- 2~3줄 요약
- 카테고리
- 지역 태그(없으면 `전국`)
- 공식발표 여부 배지
- 대표 기사 여부 또는 유사 기사 수
- 원문 링크

선택 노출 가능한 보조 메타데이터:
- `정부 원문 우선`
- `유사 보도 n건 통합`
- `청년 현실/통계`
- `논평·기고`

너무 많은 라벨로 화면이 시끄러워지지 않도록, 카드당 2개 이하의 배지만 우선 노출하고 나머지는 상세 보기에서 제공한다.

### 3.10.3 섹션별 레이아웃 원칙

- `뉴스` 페이지는 하이라이트 영역 + 카테고리 탭 + 최신 기사 목록 구조를 갖는다.
- `정책` 페이지는 공식 발표 중심으로 정렬하며, 기관명과 발표일을 눈에 띄게 표시한다.
- `허브` 페이지는 정보성 카드보다 활동 맥락을 설명하는 허브형 레이아웃을 사용한다.
- `도구` 페이지는 자료실/활용법/검토 요청의 3단 구조로 나눈다.
- `연락` 페이지는 운영 목적, 문의 유형, 응답 기대치를 명확히 설명한다.

### 3.10.4 메뉴별 기능 구현 방식

각 메뉴는 단순 페이지가 아니라, 별도의 데이터 흐름과 사용자 행동을 가진 기능 단위로 설계한다.

#### 3.10.4.1 `뉴스` 메뉴

**핵심 목표**:
- 라이트 이용자가 메인 뉴스 흐름을 빠르게 읽고 떠날 수 있어야 한다.
- 중복 기사와 반복 논조를 줄이고, 청년 현실·통계·지역 이슈를 선명하게 분리해야 한다.

**핵심 사용자 행동**:
- 오늘 중요한 기사 3~5건을 먼저 훑는다.
- `청년은 지금 / 지역 이슈 / 논평·기고 / 정책 오피셜` 카테고리로 맥락을 나눠 본다.
- 원문으로 이동하기 전에 `왜 중요한지`를 짧게 파악한다.

**v1 구현 구성**:
- 데일리 하이라이트 블록
- 카테고리별 기사 목록
- 대표 기사 1건 + 유사 기사 수 표기
- 지역 태그(광역시도)
- 카드 메타데이터: 카테고리, 지역, 출처, 날짜, 정부 원문 우선 여부

**데이터 구조**:
- 입력: 언론 RSS, 정책브리핑 청년정책 뉴스, 국무조정실 보도자료
- 처리: 중복 클러스터링 → 카테고리 분류 → 중요도 스코어링 → 10건 내 선별
- 출력 필드: `title`, `summary`, `categories`, `region`, `published_date`, `related_article_count`, `source`, `is_official_source`

**운영 개입 포인트**:
- 특정 소스 on/off
- 반복 노이즈 키워드 차단
- 대표 기사 강제 지정
- 특정 기사 상단 고정

**v2 확장 방향**:
- 날짜/지역/카테고리 필터 고도화
- 동일 이슈 묶음 상세 보기
- `왜 이 기사가 선택되었는가` 설명 노출

#### 3.10.4.2 `정책` 메뉴

**핵심 목표**:
- 정부발 오피셜 발표를 중심으로 최근 정책 변화를 추적하는 페이지가 되어야 한다.
- 뉴스 해석보다 원문 확인과 기관 단위 파악이 쉬워야 한다.

**핵심 사용자 행동**:
- 최근 발표 순으로 정책 변화를 훑는다.
- 국무조정실, 부처, 위원회 등 발행 기관 흐름을 확인한다.
- 뉴스 기사보다 먼저 정부 원문에 접근한다.

**v1 구현 구성**:
- 최신 발표 카드 목록
- 발행 기관/발표일 전면 노출
- `정부 원문 우선` 배지
- 정책 유형 표시: 보도자료, 회의 결과, 지정, 공모, 제도 안내

**데이터 구조**:
- 입력: 국무조정실, 정책브리핑, 주요 부처 공식 발표
- 분류 필드: `issuing_org`, `document_type`, `published_date`, `source_kind`
- 정렬 기준: 최신 발표일 우선, 같은 날짜면 공식성 높은 기관 우선

**운영 개입 포인트**:
- 필수 기관 화이트리스트 관리
- 문서 유형 오분류 보정
- 정책 중요도 수동 상향

**v2 확장 방향**:
- 기관별 아카이브
- 정책 변화 타임라인
- 제도별 묶음 보기 (`청년주거`, `일자리`, `금융` 등)

#### 3.10.4.3 `청년활동가 허브` 메뉴

**핵심 목표**:
- 활동가가 단순히 읽고 끝나는 것이 아니라, 거버넌스와 협업 흐름을 한 곳에서 인지하게 해야 한다.
- 연구자와 활동가가 `현장의 움직임`을 구조적으로 파악할 수 있어야 한다.

**핵심 사용자 행동**:
- 정부 거버넌스 일정/회의/발표를 본다.
- 지역 단위 청년 거버넌스나 네트워크 활동을 찾는다.
- 향후 커뮤니티/협업 제안 흐름으로 진입한다.

**v1 구현 구성**:
- 정부 거버넌스 블록
- 지역 거버넌스 블록
- 커뮤니티 예고 블록
- 활동 맥락 설명 카드 (회의, 위원회, 협약, 발표회, 모집)

**데이터 구조**:
- 입력: 제목+내용 기준으로 허브 키워드가 감지된 기사 + 수동 큐레이션
- 태그: `hub_topics`, `governance_scope(정부/지역)`, `governance_activity_types`, `status`
- 현재는 게시판형 자유글보다 `구조화된 활동 피드`를 우선한다.

**정부 거버넌스 피드 설계**:
- 수집 대상:
  - `2030자문단`, `2030청년자문단`, `청년보좌역`, `청년정책조정위원회`, `청년정책 관계장관회의`, `청년정책 실무조정회의` 등 정부가 주도한 청년 회의/위원회/협의체 기사
  - 국무조정실, 정책브리핑, 향후 주요 부처 보도자료/브리핑 기사
- 분류 조건:
  - 허브 키워드와 함께 `회의`, `간담회`, `위원회`, `협약`, `발표회`, `브리핑` 같은 활동 맥락이 함께 있을 때 피드에 편입
  - 공식 출처 우선, 동일 회의 기사 다수 발생 시 대표 기사 1건 중심으로 정리
- 화면 구성:
  - 최신 정부 거버넌스 기사 카드
  - 활동 유형 배지 (`회의`, `위원회`, `협약` 등)
  - 후속 기사 연결 또는 상단 고정 처리

**지역 거버넌스 피드 설계**:
- 수집 대상:
  - `청년협의체`, `청년위원회`, `청년네트워크`, `청년참여단`, `청년정책협의체` 등 지방자치단체 운영 거버넌스 기사
  - 광역·기초 지자체와 지역 네트워크 활동 기사
- 분류 조건:
  - 청년 관련 키워드 + 지역 거버넌스 키워드 + 활동 맥락(`회의`, `간담회`, `포럼`, `워크숍`, `모집`, `출범`)이 함께 있을 때 편입
  - `지자체`, `시청`, `도청`, `구청`, `시장`, `도지사` 등 지역 행정 맥락이나 광역시도 태그가 함께 감지되어야 함
- 화면 구성:
  - 정부 거버넌스 피드와 분리된 카드 그리드
  - 광역 단위 우선 표출
  - 지역이 분명하지 않은 기초 단위 기사는 운영자가 보정

**커뮤니티 기능 설계 (v2 준비)**:
- 시작 방식:
  - 자유 게시판이 아니라 `운영자 주제 설정형` 익명 토론으로 시작
  - 운영자가 특정 정책 주제와 질문 범위를 직접 설정
- 사용자 흐름:
  - 주제 공지 확인
  - 익명 의견 제출 (`경험`, `찬반`, `보완 제안`, `현장 사례`)
  - 운영자 요약 브리핑 확인
- 운영 원칙:
  - 정치 성향 대결형 구조보다 `논점 수렴형` 구조를 우선
  - 익명성은 허용하되, 욕설/비방/도배는 운영자가 비공개 처리
  - 토론 결과는 기사/정책 흐름과 연결 가능한 요약 기록으로 환류

**운영 개입 포인트**:
- 허브 연관 기사 선별
- 이벤트/활동 종료 처리
- 중요 활동 상단 고정
- 지역 거버넌스 기사 태그 보정
- 커뮤니티 주제 개설/종료 및 요약 게시

**v1/v2 경계**:
- v1: 읽기 중심 허브 + 구조화된 활동 피드
- v2: 커뮤니티, 협업 제안, 정책 제안 공유 흐름 도입

#### 3.10.4.4 `정책제안서 작성 도구` 메뉴

**핵심 목표**:
- 자동 대필기가 아니라, 정책을 찾고 정리하고 검토받는 실무 보조 도구가 되어야 한다.
- 헤비 사용자와 활동가의 정책 제안 품질을 높이는 것이 목표다.

**핵심 사용자 행동**:
- 정책 자료 출처를 찾는다.
- AI를 어디까지 써도 되는지 가이드를 본다.
- 초안 검토 요청 또는 협업 요청 흐름으로 이동한다.

**v1 구현 구성**:
- 정책 서칭 주요 사이트 모음
- AI 활용 가이드
- 정책제안서 템플릿/가이드
- 검토 요청 준비 체크리스트

**데이터 구조**:
- 정적 문서 중심 (`proposal_template`, `writing_guide`, `data_sources_guide`)
- 링크 목록은 목적별로 묶는다: 정책 원문, 통계, 입법/예산, 지역 자료

**운영 개입 포인트**:
- 참고 링크 추가/비활성화
- AI 사용 가이드 문구 업데이트
- 템플릿 버전 관리

**v2 확장 방향**:
- 인터뷰형 질문 도구
- 논점 정리 보조
- 목차 초안 생성
- 검토 요청 접수 양식 연결

#### 3.10.4.5 `운영자 연락` 메뉴

**핵심 목표**:
- 문의, 제보, 협업, 검토 요청이 섞이지 않도록 운영 접점을 명확히 분리해야 한다.
- 운영자의 대응 범위와 기대 응답을 투명하게 설명해야 한다.

**핵심 사용자 행동**:
- 버그/사용성 문의를 보낸다.
- 지역 활동이나 유용한 소스를 제보한다.
- 정책제안서 검토 요청 경로를 확인한다.

**v1 구현 구성**:
- 문의 유형 분류 카드
- 제보/협업 제안 안내
- 검토 요청 연결 안내
- 응답 기대치와 운영 범위 설명

**데이터 구조**:
- 현재는 정적 안내 페이지 + 외부 폼 연결 전 단계
- 향후 접수 데이터는 `inquiry_type`, `contact_channel`, `summary`, `status`로 구조화 가능

**운영 개입 포인트**:
- 문의 유형 문구 수정
- 응답 우선순위 정책 공지
- 연결 채널 교체

**v2 확장 방향**:
- 외부 폼 또는 Notion/이메일 접수 연결
- 문의 상태 추적
- 제보 → 소스 등록 검토 흐름 연결

|------|------|------|
| 정책 제안서 템플릿 | Markdown + DOCX | 제목, 현황 분석, 문제 정의, 제안 내용, 기대 효과, 참고자료 섹션 |
| 작성 가이드라인 | Markdown | 각 섹션별 작성 팁, 좋은/나쁜 예시, 분량 가이드 |
| 데이터 소스 가이드 | Markdown | 통계 데이터 출처 (KOSIS, 지자체 공공데이터 등), 검색 방법, 인용 형식 |

### 4.2 설계 원칙

- v1에서 정책 제안서 도구는 **대화형 자동 작성기**가 아니라 템플릿·가이드·참고자료 제공 도구로 제한한다.
- 사용자의 사고를 돕는 방향을 우선하며, 자동으로 본문 전체를 대필하는 기능은 도입하지 않는다.
- 향후 확장 시에도 `질문 던지기`, `근거 정리`, `목차 초안`, `문장 다듬기` 수준의 보조 기능을 우선 검토한다.
- LLM이 특정 정치적 방향성이나 정답을 제시하는 인상을 주지 않도록, 출처 기반 보조도구로 설계한다.

### 4.3 향후 확장 방향

- v2: 인터뷰형 질문 도구, 근거 정리, 제안서 목차 초안 생성
- v2: 연구자용 커뮤니티/현장 반응 탐색 기능 검토
- v3: 플랫폼에 축적된 기사 데이터를 근거 자료로 자동 연결

> `본문 전체 초안 대필`은 장기 검토 항목으로 분리하며, 기본 방향은 비도입에 가깝게 유지한다.

운영자는 분류 결과를 대량으로 수동 라벨링하지 않는 대신, 아래의 제한된 개입 지점을 통해 품질을 조정한다.

| 개입 포인트 | 설명 | 빈도 |

| 소스 on/off | 특정 RSS, 정책 브리핑, 지역 매체 소스를 활성/비활성화 | 수시 |
| 대표 기사 강제 지정 | 유사 기사 묶음에서 노출할 대표 기사를 수동 지정 | 필요 시 |

- 프롬프트 튜닝은 허용하되, 매일 수동 검수·라벨링이 필요한 설계는 피한다.

- 사람 손이 자주 가야 유지되는 규칙은 v1에서 도입하지 않는다.

## 부록 D: 항상 켜짐형 운영 배포 원칙

로컬 PC 기반 작업 스케줄러는 개발/테스트에는 적합하지만, 운영 기준의 자동 갱신 구조로는 한계가 있다. 운영 환경에서는 `항상 켜져 있는 Linux VM`에서 정해진 시각마다 자동 실행되는 구조를 기본값으로 삼는다.

### 권장 운영 구조

```text
systemd timer (09:00 / 15:00 / 21:00, Asia/Seoul)
        │
        ▼
run_pipeline.sh
        │
        ▼
cron_runner.py
        │
        ├─ output/*.json
        ├─ db/articles.db
        └─ web/*.html
        │
        ▼
nginx 정적 서빙
```

### 운영 원칙

- 로컬 배치가 아니라 `클라우드 VM 상시 운영`을 기본으로 한다.
- 배치 실행 시각은 `09:00 / 15:00 / 21:00 (Asia/Seoul)`를 유지한다.
- 서버 재부팅이나 일시 중단 후에도 놓친 실행을 보완할 수 있도록 timer의 `Persistent=true`를 사용한다.
- 중복 실행은 파이프라인 lock 파일과 systemd 단일 실행 원칙으로 차단한다.
- 상태 파일(`pipeline_status.json`)과 로그 파일(`output/scheduler_logs/`)을 함께 보관한다.

### 기술 선택 이유

- 현재 프로젝트는 `정적 웹 결과물 + SQLite + Python 배치` 구조라서, 별도 API 서버보다 `단일 VM + nginx`가 구현 비용이 가장 낮다.
- 추후 트래픽 또는 데이터가 커지면 다음 순서로 확장한다.
  - 배치 worker와 웹 서빙 분리
  - SQLite에서 PostgreSQL로 전환
  - 정적 파일을 CDN/정적 호스팅으로 이전

### 운영 체크 항목

- 마지막 성공 실행 시각
- 다음 예약 실행 시각
- 마지막 종료 코드
- 수집 기사 수 / 최종 발행 기사 수
- 최근 7일 연속 실패 여부
